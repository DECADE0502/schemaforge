# SchemaForge 重启说明（2026-03-08）

## 文档目的

这份文档不是对现有仓库的粉饰，也不是“继续修补当前代码”的 TODO 列表。

它的目的只有一个：

> 如果今天决定重启 SchemaForge，这个项目到底应该被定义成什么，核心难点是什么，第一版应该如何收口。

这份文档代表的是我对 **SchemaForge 最终目标、正确边界、核心工程难题、重启策略** 的统一理解。

---

## 一句话定义最终目标

SchemaForge 的最终目标应该是：

> 用户用自然语言描述一个电子系统，系统自动理解需求，精确识别器件，必要时引导补录缺失器件，然后基于器件知识、规则、公式和连接约束，生成一张真实可读的系统级原理图，并同时导出 BOM、SPICE，以及支持多轮增量修订。

这里有三个关键词必须抓死：

1. **系统级**：不是单模块示意图，而是一整套可连接、可导出的系统图。
2. **精确型**：用户指定的料号必须被当成硬约束，不允许“差不多”。
3. **工程化**：外围件、连接、参数和输出必须有依据，不能靠模板拼接和拍脑袋默认值伪装成自动设计。

---

## 项目真正要解决的问题

从本质上讲，SchemaForge 不是“画图项目”，而是一个由 7 个问题串起来的系统：

1. **需求理解**
   - 从自然语言中识别模块、器件、目标电压、电流、控制关系、上下游供电关系。

2. **器件识别**
   - 把 `TPS54202`、`STM32F103C8T6`、`AMS1117-3.3` 这种具体料号解析成确定对象。
   - 如果用户说的是显式料号，必须先做 exact match，不允许退化成类别匹配。

3. **器件补录**
   - 本地没有的器件不能被偷换，必须进入缺料流程。
   - 用户提供 PDF / 图片后，要能提取引脚和关键参数，形成结构化器件记录。

4. **模块综合**
   - 对电源、LDO、LED、MCU 最小系统等模块生成外围件、参数、网络角色和约束。
   - 这一步必须依赖 datasheet / 规则 / 公式，不应只是套模板。

5. **系统连接**
   - 模块之间的连接必须是电气真实连接，而不是把几个方块摆在一张图里。
   - 系统要能建立共享网络、上下游电源链、GPIO 控制链。

6. **系统渲染**
   - 把系统 IR 和布局状态变成一张真实可读的系统图。
   - 这一步必须支持布局状态输入，否则 visual review 永远是假闭环。

7. **增量修订**
   - 文本修订、图片修订、补录器件后的继续设计，都必须回到同一个会话对象和同一套系统状态上。

只要这 7 个问题没有串成一条真链路，项目就不能说完成。

## 最低正确输出长什么样

下面这句话，应该被当作 SchemaForge 的最小金路径：

> `20V 输入，用 TPS54202 降压到 5V，再用 AMS1117 降到 3.3V，给 STM32F103C8T6 供电，并且用 MCU 的 PA1 控制一颗 LED`

一个合格的系统必须至少输出：

- `TPS54202` Buck 模块及其外围
- `AMS1117-3.3` LDO 模块及其外围
- `STM32F103C8T6` 的最小供电连接
- `LED + 限流电阻`
- `Buck.VOUT -> LDO.VIN`
- `LDO.VOUT -> MCU.VCC`
- `MCU.PA1 -> LED`
- 单张系统 SVG
- 合并后的 BOM
- 合并后的共享网络 SPICE

如果连这条都做不真，就不应该谈“通用化”。

## 必须坚持的硬约束

### 1. 料号精确优先

这是整个项目最关键的原则之一：

- 用户说 `TPS54202`，就处理 `TPS54202`
- 用户说 `TPS5450`，而库里没有 `TPS5450`
- 正确行为应该是：**返回缺料 / 需要补录**
- 错误行为是：擅自退化成 `TPS5430`、`TPS54202` 或任意同类 Buck

换句话说：

> “类别正确但型号不对” 在这个项目里不是小误差，而是原则性失败。

### 2. AI 只负责理解和提取，不负责替代工程引擎

AI 的正确职责是：

- 提取模块意图
- 理解连接关系
- 识别器件名和料号
- 从 datasheet / 图片中提取结构化信息
- 生成 revision instruction

AI 不应该直接负责：

- 自己发明引脚映射
- 自己发明外围参数
- 自己决定网络连接是否合理
- 自己随意输出原理图内容

这些必须落在本地、可验证、可测试的规则和数据结构上。

### 3. 缺料流程必须是一等公民

“器件库不全” 不是异常边角，而是主流程的一部分。

所以系统必须天然支持：

- 精确命中失败
- 向用户清楚说明缺的是哪个料号
- 引导上传 PDF / 图片
- 提取引脚、封装、关键参数
- 生成结构化记录
- 用户确认后入库
- 回到原设计继续跑

### 4. 渲染器必须消费布局状态

只要 renderer 不接受并消费 `layout_spec` 之类的真实布局状态对象：

- visual review patch 就是改空气
- 图片反馈就只是表演
- 布局优化就无法闭环

因此：

> `render_system_svg(ir, layout_spec) -> RenderResult` 应该是重启后的硬接口之一。

### 5. GUI / CLI / revise 必须共用一条后端链路

不能再出现：

- GUI 一套会话
- CLI 一套会话
- 图片 revise 一套会话
- 文本 revise 一套会话

重启后必须只有一个系统级主会话：

> `SystemDesignSession`

所有入口都只是它的适配器。

## 为什么这个项目难

这个项目看起来像“AI 画电路”，但真正难点并不在“画图”，而在下面这几个组合问题。

### 1. 需求语言和电气语义之间不是一一映射

用户会说：

- “给 MCU 供电”
- “再降到 3.3V”
- “用 PA1 控制 LED”
- “多加一个 1.8V 支路”

这些句子不是直接等于原理图对象，而是需要映射成：

- 模块集合
- 模块属性
- 连接意图
- 控制关系
- 优先级和约束

因此系统最难的第一步不是 LLM 调一下，而是定义稳定的 **系统意图 schema**。

### 2. 精确型器件识别和类别推断天然冲突

一个好的自然语言系统往往喜欢“尽量理解用户意思”，但电路设计不能这样。

- NLP 倾向于模糊容错
- EDA 倾向于精确命中

SchemaForge 必须明确站在 EDA 一边：

- 显式料号 -> 精确匹配
- 类别描述 -> 类别匹配
- family 变体 -> 只能在证据唯一时推断
- 否则就返回缺料

### 3. 模块级正确，不等于系统级正确

单独把 Buck、LDO、LED、MCU 都生成出来不难。

真正难的是：

- 电源链路连对
- 网络名共享正确
- GPIO 接口方向合理
- BOM 编号全局不冲突
- 渲染结果在一张图上还能读

所以系统重心不应该放在“模块越来越多”，而应放在：

> 模块如何进入统一系统 IR 并被统一连接、统一渲染、统一导出。

### 4. 渲染不是附件，而是核心域模型的一部分

如果渲染器只是从最终对象“随便摆一下”：

- 视觉反馈无法精确修改布局
- 布局质量不可测
- 复杂系统会迅速变得不可读

这意味着布局状态必须成为正式输入，而不是 renderer 内部临时变量。

### 5. revise 不是重新 start 一遍

多轮修改的难点在于：

- 需要知道用户改的是哪个模块
- 是替换器件、改参数、加模块、删模块，还是改连接
- 需要保持已有 reference、网络名、模块关系尽量稳定

如果 revise 只是把用户新句子再跑一次生成新图：

- 很容易丢状态
- 很容易重复创建模块
- 很容易造成“看起来改了，实际上换了一套图”

### 6. 图片反馈只有在“图形对象可寻址”时才有意义

图片审稿之所以难，不在于 OCR 或 vision，而在于：

- 图片里的“这个模块太靠左”要映射到哪个 module_id
- “这根线挡住文字了”要映射到哪个 edge / segment
- patch 应该修改哪个 layout field
- 修改后 renderer 是否真的会消费它

因此图片 revise 不是单独功能，而是 **布局状态可寻址性** 的副产品。

## 如果重启，正确的主架构应该是什么

我建议重启后只保留一条主链，严格按下面的层次设计。

### A. Intent 层

输入：自然语言 / 图片 / 补录结果

输出：

- `SystemDesignRequest`
  - `modules[]`
  - `connections[]`
  - `constraints[]`
  - `revision_ops[]`（修订时）

目标：只做“理解成结构化请求”，不做工程决策。

### B. Resolver 层

输入：`SystemDesignRequest`

输出：

- 每个模块的器件解析结果
  - exact resolved
  - family resolved
  - missing / needs_asset

目标：处理料号解析、类别解析和缺料判定。

### C. Synthesis 层

输入：解析后的模块

输出：

- 模块内部元件
- 参数计算结果
- 端口暴露
- 网络角色

目标：把“模块意图”变成“可接入系统”的模块实例。

### D. System IR 层

输入：多个模块实例 + 连接规则

输出：

- 统一系统图结构
- 模块实例
- 连接边
- 共享网络
- 导出所需信息

目标：这是整个系统的单一事实源。

### E. Layout 层

输入：`SystemIR`

输出：`LayoutSpec`

包括：

- module positions
- port anchors
- edge routes
- labels / bbox
- spacing constraints

目标：让“图长什么样”成为正式状态对象。

### F. Rendering 层

输入：`SystemIR + LayoutSpec`

输出：

- SVG
- `render_metadata`

目标：只负责把确定的图形状态画出来，不自己偷偷决定布局逻辑。

### G. Export 层

输入：`SystemIR`

输出：

- BOM
- SPICE
- 审查信息

目标：保证导出来自同一个系统事实源，而不是各导各的。

### H. Session 层

职责：

- `start()`
- `revise()`
- `revise_from_image()`
- `resume_after_ingest()`

目标：统一状态、统一入口、统一修订。

## 重启时第一版应该故意砍掉什么

为了避免项目再次走向“结构很多但没有真闭环”，第一版应主动砍掉这些东西：

1. **通用 visual review 自动迭代**
   - 在 renderer 真吃 `LayoutSpec` 之前，不应该做。

2. **花哨 GUI 状态和按钮**
   - 先把主链做真，再补 UI 体验。

3. **过宽的器件支持面**
   - 第一版先只做 Buck / LDO / MCU power / LED / GPIO。

4. **多种并存的会话体系**
   - 只留一个系统级 session。

5. **大而全的 mock 测试堆积**
   - 第一版测试重点是金路径和缺料路径，而不是函数覆盖率表演。

---

## 第一版真正该做的最小范围

只支持下面这四类对象：

- Buck
- LDO
- MCU power + GPIO pin exposure
- LED indicator

只支持下面四类操作：

- 新建设计
- 替换已有器件
- 改已有参数
- 增删 LED / 下游 LDO

只支持下面三类输出：

- 单张 SVG
- 全局 BOM
- 全局 SPICE

只支持下面三类异常：

- 缺料
- 歧义目标
- 连接无法满足

如果第一版能把这些做真，项目就有可持续扩展的基础。

## 建议的阶段性验收标准

### Phase 1：金路径打通

验收问题：

1. 输入金路径长句后，是否稳定得到 4 个模块？
2. 是否稳定得到 3 条关键连接？
3. 是否生成单张系统 SVG？
4. BOM 是否包含 4 类核心器件？
5. SPICE 是否共享 `5V / 3.3V / GPIO` 网络？

### Phase 2：缺料路径打通

验收问题：

1. 输入不存在的明确料号时，是否返回缺料而不是替代？
2. UI 是否能清楚显示缺的是哪个料号？
3. 补录后是否能继续原设计，而不是从零开始？

### Phase 3：文字 revise 打通

验收问题：

1. `把 TPS54202 换成 TPS5430` 是否替换现有模块，而不是新增？
2. `把 5V 改成 4.2V` 是否命中正确模块？
3. `再加一个 PA2 控制的蓝色 LED` 是否增量加到现有系统图？

### Phase 4：布局状态闭环

验收问题：

1. renderer 是否真实消费 `LayoutSpec`？
2. patch 是否能改变 SVG 输出？
3. 导出结果是否不被布局修改破坏？

在这之前，不要宣称图片 revise 或 visual review “已完成”。

## 测试策略应该怎么改

重启后，测试应该围绕“真实用户路径”，不是围绕“模块存在”。

推荐测试结构：

1. **金路径集成测试**
   - 一条真实系统需求 -> IR / SVG / BOM / SPICE

2. **缺料路径测试**
   - 显式料号不存在 -> needs_asset

3. **修订路径测试**
   - 替换器件 / 改参数 / 加模块 / 删模块

4. **渲染闭环测试**
   - 同一 IR，不同 LayoutSpec，SVG 不同但网络一致

5. **最少量 source-level wiring 测试**
   - 确保 GUI / CLI 都接在同一 session 上

不建议把大量精力继续花在：

- mock AI 调用次数
- 旧路径兼容细节
- 纯 dataclass 存在性
- 不走主链的子模块局部函数表演

---

## 重启时最需要避免的坑

### 坑 1：把“结构更完整”误当成“链路更真实”

多几个 dataclass、多几个 planner、多几个 patch executor，并不会让项目更接近可用。

### 坑 2：太早做通用化

先把 `Buck -> LDO -> MCU -> LED` 做真，再谈 sensor、opamp、USB、通信接口。

### 坑 3：AI 权责失控

一旦 AI 开始越过本地规则直接替你决定工程内容，系统会很快变得不可验证。

### 坑 4：渲染器继续当黑箱

只要布局状态不是正式输入，后面所有“视觉闭环”都会再次变成假闭环。

### 坑 5：入口分叉

如果从一开始就没有压死为一个统一主会话，后面一定再次分裂出 GUI / CLI / image 各自的链路。

## 我对重启项目的建议结论

如果这个项目要重启，最重要的不是“换个模型”或者“多写一点代码”，而是先统一这 5 条共识：

1. **目标是系统级自动设计，不是单器件绘图。**
2. **显式料号必须精确命中，缺料就缺料。**
3. **AI 负责理解，本地规则负责工程决策。**
4. **布局状态必须成为 renderer 的真实输入。**
5. **GUI / CLI / revise / ingest 必须共用一个主会话。**

只要这 5 条不变，项目就能被重新做成一个真实系统。

如果这 5 条不先定死，SchemaForge 很容易再次滑回：

- 功能越来越多
- 文档越来越乐观
- 测试越来越多
- 真实主链仍然不可信

## 建议的重启口号

最后我给这个项目一个更准确也更有约束力的定义：

> SchemaForge 不是一个“会画几张电路图的 AI Demo”，而应该是一个“由自然语言驱动、以精确器件和系统 IR 为核心的系统级原理图生成引擎”。

如果以后任何设计、实现、测试、文档改动和这句话相冲突，就应该优先怀疑改动，而不是怀疑目标。
