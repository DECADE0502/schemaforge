# SchemaForge 项目记录

这个文档记录我做 SchemaForge 这个项目的完整过程，从最初的想法到最后决定暂停。写这个主要是给自己留个档，以后回头看能知道当时怎么想的、踩了什么坑、学到了什么。


## 背景

我想做一个 AI 驱动的电路原理图生成工具。核心想法很简单：用户说一句自然语言（比如"用 TPS5430 搭一个 12V 转 3.3V 的 DCDC"），系统自动输出完整的原理图 SVG、BOM 清单和 SPICE 网表。

之所以想做这个，是因为画原理图这件事本质上是高度规则化的：选型有 datasheet 约束，外围计算有公式，连线有电气规则，布局有工程惯例。理论上 AI 理解需求之后，剩下全是确定性计算，应该能自动化。

工具选型：Python + PySide6 做 GUI，schemdraw 做原理图渲染，kimi-k2.5 做 LLM（通过 DashScope 的 OpenAI 兼容接口调用），Pydantic 做数据模型。


## 阶段一：需求拆解和架构设计

一开始我花了不少时间想清楚这个系统应该怎么分层。核心原则只有一条：AI 只做理解和决策，所有计算、渲染、约束检查都是本地确定性代码。

这个原则不是凭空来的。之前试过让 AI 直接生成 schemdraw 代码，结果发现它经常搞错坐标、漏连线、元件重叠。所以从一开始就决定 AI 的输出只能是结构化数据（JSON），然后本地代码根据这些数据做所有后续工作。

架构拆分大概是这样：

- AI 层：自然语言 -> 结构化意图（器件型号、电压参数、连接关系）
- 器件库层：器件数据存储、查询、PDF 入库
- 设计层：连接规则引擎、参数综合（公式计算外围元件值）、工程审查（42 条规则）
- 渲染层：确定性 SVG 生成
- 导出层：BOM + SPICE

每一层之间通过 Design IR（中间表示）连接，这个 IR 是整个系统的唯一真值。上游写 IR，下游读 IR，谁都不能绕过去。

在具体实现之前，我先把 task 拆得很细。比如器件库这块：

1. 定义 DeviceModel 数据模型（引脚、参数、拓扑、设计公式）
2. 实现 JSON + SQLite 双重存储（SQLite 做索引搜索，JSON 存完整数据）
3. PDF datasheet 提取管线（PyMuPDF 提文本 -> AI 提取结构化数据 -> 校验 -> 入库）
4. 给每个环节写测试

这种拆法的好处是每个 task 都有明确的输入输出和验收标准，做完一个验一个，不会做到后面发现前面有问题。


## 阶段二：v1/v2 迭代和推翻

最开始的 v1 架构比较简单：一个 engine.py 做核心调度，render/ 目录下每种电路一个渲染器（ldo.py、led.py、divider.py 等），workflows/ 里放状态机和会话管理。

v1 能跑单器件的情况——给一个 LDO 型号，能出一张原理图。但问题很快暴露：

- 每种电路类型都要写一个专用渲染器，扩展性差
- 单器件会话无法处理多模块系统（比如 Buck + LDO + MCU 的电源链）
- 旧状态机太僵硬，改个参数要重跑整个流程

所以做了 v2，引入了 IR patch 机制，想通过增量修改来支持多轮对话。但 v2 很快也撞墙了——IR patch 的粒度很难控制，改一个电压值可能需要联动改好几个元件的参数，patch 引擎写得越来越复杂。

最后决定推翻重来，直接做 v3 的 AI Agent 架构。这个决定现在回头看是对的，但当时确实犹豫了一阵——前面写的几千行代码全要扔掉。


## 阶段三：v3 AI Agent 架构

v3 的核心思路是把 AI 从"一次性生成"变成"多轮工具调用"。用 OpenAI function calling 协议，定义了 13 个原子工具，AI 在每一轮对话中可以调用任意工具，系统执行工具并返回结果，AI 再决定下一步。

工具设计的时候花了很多心思在 prompt 构建上。system prompt 要清楚告诉 AI：

- 你有哪些工具可用，每个工具的输入输出是什么
- 正确的调用顺序是什么（先查库 -> 解析模块 -> 解析连接 -> 综合参数 -> 渲染 -> 导出）
- 什么时候该问用户，什么时候该自己决定
- 不要编造器件参数，所有参数必须从器件库获取

prompt 迭代了很多版。最开始 AI 经常跳步骤，比如还没查库就直接开始解析模块；后来加了明确的步骤约束和错误处理指令才稳定下来。

这块的 task 拆分：

1. Orchestrator 核心循环（function calling 解析、工具分发、结果回收）
2. 13 个工具逐个实现（每个工具都是纯函数，输入 JSON 输出 JSON）
3. SystemDesignSession 状态管理（start/revise/ingest/confirm 四个主动作）
4. System prompt 调试（反复测试不同的 prompt 策略）
5. 多轮修订支持（revise 管线要能处理"把输出改成 5V"、"换一个料号"、"加一颗 LED"这类修改）

v3 的效果确实好很多。能处理 "20V 输入，用 TPS54202 降压到 5V，再用 AMS1117 降到 3.3V，给 STM32 供电，PA1 控制一颗 LED" 这种多模块链式需求，输出的 SVG 包含完整的电源链和信号连接。


## 阶段四：SVG 渲染问题 —— 核心困难

这是整个项目最卡的地方。

在自己写渲染器之前，认真试过两个现成的方案：KiCad 和嘉立创 EDA（LCEDA/EasyEDA）。花了不少时间，最后都放弃了。

KiCad 方案——试了，出图效果不行：

KiCad 从 6.0 开始 .kicad_sch 文件改成了 S-expression 文本格式，理论上可以直接拼文件生成原理图。我实际试了这条路，写了一个原型去生成 .kicad_sch，把我们的 Design IR 翻译成 KiCad 的符号放置 + 连线。踩到的具体问题：

第一个问题是符号库绑定。KiCad 原理图里每个元件不是自包含的，它引用一个 .kicad_sym 库文件里的符号定义。也就是说你要在 .kicad_sch 里放一个 TPS54202，对应的 .kicad_sym 里必须有这个符号的完整定义（矩形框、引脚位置、引脚名、引脚编号、图形元素）。我们的器件库用的是自己的 SymbolDef 格式（按 KLC 标准的引脚分边+自动布局），要转成 KiCad 的 .kicad_sym 格式需要处理一堆细节：引脚的 at 坐标要用 KiCad 的 mil 坐标系、图形元素要用 KiCad 的 polyline/rectangle 语法、文字要设置正确的 font size 和 justify。我写了转换代码，但出来的符号在 KiCad 里打开，引脚名称和引脚编号经常重叠，文字位置偏移，因为 KiCad 的文字渲染和我计算的 bounding box 不完全一致。

第二个问题是连线。KiCad 原理图的连线（wire）是用绝对坐标的 (x1,y1)-(x2,y2) 线段。问题在于你必须保证线段端点精确落在引脚端点上，差 1mil 都算没连上，KiCad 的 ERC 会报 unconnected pin。而且 KiCad 的引脚端点位置取决于符号定义里 pin 的 at 坐标加上符号放置时的 at 坐标和旋转变换，算起来很容易出错。我的原型跑出来的 .kicad_sch 在 KiCad 里打开，大概有 30% 的连线看起来连上了但实际上 ERC 报错，因为坐标差了零点几 mil。手动在 KiCad 里放大看，线段端点和引脚之间有一个肉眼看不到的缝隙。

第三个问题是部署。要验证生成的 .kicad_sch 是否正确，必须装 KiCad（Windows 安装包 2G+）。而且 KiCad 的 Python 脚本接口（pcbnew）只覆盖 PCB 端，eeschema 没有对应的 Python API，没法在 CI 里自动化验证生成的原理图是否合法。只能靠人肉打开 KiCad 看。这对一个想做"自动化生成"的工具来说是不可接受的——你生成了一个文件，但没法自动验证它是不是对的。

最后评估下来，即使投入更多时间把这些坐标精度问题修好，最终用户还是得装 KiCad 才能看到结果，而且生成的原理图只能在 KiCad 里编辑。不如直接输出 SVG——浏览器就能看，不绑任何软件。

嘉立创 EDA（LCEDA/EasyEDA）方案——API 只有器件查询，没有原理图绘制：

嘉立创这边我们实际用了两个 API，都已经在代码里跑通了（easyeda_provider.py）：

1. JLCPCB 搜索 API（jlcpcb.com/api/...selectSmtComponentList）——输入型号关键字，返回器件列表，包含 LCSC 编号、封装、价格、库存。这个用来做器件搜索入口。
2. EasyEDA 产品 API（easyeda.com/api/products/{lcsc_id}/components）——输入 LCSC 编号，返回器件的原理图符号数据，包括引脚定义（name、number、electrical type、坐标）和符号图形（shape 数组里的 polyline、rectangle 等）。这个用来补全器件引脚信息，比人工填引脚快很多。

但这两个 API 都是器件级别的——只能查"一个元器件长什么样"，不能操作"一张原理图"。

EasyEDA 的原理图编辑器本身是个 Web 应用，编辑操作走的是 WebSocket + 私有协议，不是 REST API。它没有公开的"创建原理图项目""在原理图上放置元件""画连线"这类接口。我翻过 EasyEDA 的开发者文档（https://docs.easyeda.com/en/API/），里面只有元件库相关的 API，没有原理图编辑 API。也看过 EasyEDA 的开源编辑器代码（easyeda2kicad 这个第三方工具），它只能从 EasyEDA 导出到 KiCad，不能反过来。

LCEDA Pro（嘉立创 EDA 本地版）更封闭。它是 Electron 套壳应用，没有命令行接口，没有插件系统，没有脚本 API。唯一能自动化的方式大概是用 Selenium 或者 Playwright 去操控它的 UI，这不现实。

我还想过一个方案：直接生成 EasyEDA 的项目 JSON 文件，然后用"导入"功能打开。EasyEDA 的原理图文件格式我看了一下，它有一个内部 JSON 格式（dataStr 字段），里面用 shape 数组描述所有图形元素。理论上可以手动拼这个 JSON。但这个格式没有正式文档，shape 数组里每个元素都是一个用 ~ 分隔的字符串（比如引脚是 "P~...~pin_name~...~x~y~rotation~..."），字段含义只能从开源转换工具里反推。而且 EasyEDA 的坐标系原点在左上角、Y 轴向下、单位是 10mil（所谓 "dot"），跟我们的 IR 坐标系完全不一样。试着拼了一个简单的 LDO 电路的 JSON，导入 EasyEDA 后元件位置对了但连线丢了一半——因为连线的 JSON 格式跟我推测的不完全一致，有些隐藏字段没填对。

最终结论：嘉立创的 API 只适合做器件数据源（我们确实在用），不适合做原理图输出。如果嘉立创以后开放原理图编辑 API，这条路可以重新评估。

所以最后的结论是：原理图渲染这块只能自己做。用 schemdraw 做基础渲染（它就是一个 Python 画电路图的库，轻量够用），再在上面叠加自己的布局引擎。

schemdraw 模式能跑，但出来的图质量一般，元件布局是堆叠式的，不太像真正的原理图。我想做一个更好的渲染器，让 AI 参与布局决策，生成更接近人类手绘风格的原理图。

一开始的想法是让 AI 生成带坐标的 SVG 模板，本地代码填充器件符号。具体流程：

1. 本地代码先根据设计 IR 生成一个"坐标模板"（每个器件的位置、引脚坐标）
2. AI 审查这个模板，提出布局改进建议
3. 本地代码根据 AI 建议调整坐标
4. 最终确定性渲染

但实际测试下来，即使给了 AI 完整的坐标模板，让它只做微调，结果还是不行。核心问题就两个字：重叠和断开。

- 重叠：元件之间距离不够，body 互相穿透，引线交叉
- 断开：导线应该连到引脚端点但差了几个像素，看起来就是断的

试了很多 prompt 策略：

- 给 AI 明确的坐标约束（"所有器件 x 坐标必须是 50 的整数倍"）
- 给 AI body_bbox 信息让它避让（"导线不能穿过任何 body_bbox"）
- 让 AI 先输出布局方案再输出坐标（分两步走）
- 用 vision API 让 AI 审查渲染结果并修正

每种策略都在某些 case 上有改善，但没有任何一种能做到"稳定不出错"。AI 对空间坐标的理解本质上是模糊的——它知道"不应该重叠"，但不能精确计算两个矩形是否重叠。这不是 prompt 能解决的问题，是当前 LLM 的能力边界。

最后的决定是完全放弃让 AI 参与坐标计算，改成纯本地确定性渲染。这就是 Pin + Body 架构：

- 每个器件有 pins（端点坐标）和 body（包围盒）
- 导线只能连 pin 到 pin
- 导线路由自动避让所有 body
- 布局坐标全部硬编码（Buck 一套、LDO 一套）
- Junction dots 自动检测 T 型交叉点

这套方案确实解决了重叠和断开的问题，出来的图质量也不错。但代价是每种新拓扑都需要手写一套布局代码——Buck 大概 300 多行，LDO 200 行左右。如果要覆盖 Boost、Flyback、MCU 最小系统等等，工作量很大。


## 阶段五：其他模块的推进

除了渲染，其他模块也在并行推进。

PDF Datasheet 持久化：入库的时候把用户上传的 PDF 存一份到 store/datasheets/，路径写进 DeviceModel。设计的时候 AI 可以通过 get_device_datasheet 工具读取 PDF 文本内容，这样 AI 在选外围元件参数时能参考 datasheet 的推荐值。这个功能实现起来不复杂，但对设计质量有实质帮助。

修订渲染一致性：发现 revise 的时候如果当前是 AI SVG 模式，但 _regenerate_outputs 总是走 schemdraw 渲染器，导致修订后图的风格突然变了。修复方法是给 SystemDesignSession 加了 ai_svg_mode flag，_regenerate_outputs 根据这个 flag 分发到对应渲染器。

拓扑排序 bug：layout.py 里的 _topo_sort_power 在遇到循环依赖时会无限递归。加了 in_progress 集合做灰节点检测就修好了。这种 bug 在简单测试里不出现，多轮对话加到第三四轮时才触发。

测试维护：清理完遗留代码后，很多测试文件里的用例依赖已删模块。不能一删了事，因为同一个文件里还有测试活跃模块的用例。只能逐个文件看，把依赖已删模块的用例剔掉，保留独立有效的。最后从 1566 个测试降到 1332 个，0 failures。

最后做了一轮完整的代码清理，删掉了大约 11700 行遗留代码（15 个模块、8 个测试文件、一整个无关的 flow/ 目录），更新了所有文档里过时的引用。


## 为什么决定暂停

做到这里，系统能跑，测试全过，Buck 和 LDO 的原理图输出质量也可以。但我越来越觉得继续做下去的性价比不高，主要原因：

1. 渲染器的扩展成本太高。每种新拓扑要手写几百行布局代码，而且这些代码高度 specific，不同拓扑之间没什么复用。如果有一天想支持 20 种电路类型，那就是上万行手工布局代码，这已经不是"AI 驱动"了。

2. AI 在空间推理上的能力瓶颈是根本性的。我在 SVG 渲染这块投入了大量精力尝试各种方案——坐标模板、约束 prompt、vision 审查、两阶段生成——最后的结论是当前的 LLM 做不到可靠的 2D 空间布局。它能理解"不应该重叠"这个概念，但不能一致地产出不重叠的坐标。这个问题不会因为 prompt 写得更好而解决，需要模型本身在空间推理能力上有质的提升。

3. 多模块自动布局是个更难的问题。目前的布局是单模块的（一个 Buck 一张图），如果要在一张大图上排布多个模块并自动走线，那本质上就是在做一个简化版的 EDA 布线引擎。这已经超出了一个 side project 的范围。

4. 工程审查的深度有限。42 条规则能覆盖常见的低级错误（电容耐压不够、电感饱和等），但真正有价值的审查（比如 EMI 路径分析、热设计验证）需要领域专家级的知识，目前的实现只是皮毛。


## 可能的改进方向

虽然我暂停了，但这个方向不是死路。如果以后要继续，我觉得有几个关键突破点：

关于渲染：应该放弃"从零布局"的思路，改成基于模板的参数化方案。维护一个电路模板库（手工画好的参考布局），新设计只是在模板基础上替换器件参数和调整外围元件数量。这样布局质量有保证，扩展成本也可控。KiCad 的 hierarchical sheet 就是类似思路。

关于 AI 空间推理：等模型能力上来。目前（2026 年初）的 LLM 在精确数值计算和 2D 空间推理上都不可靠。也许未来某一代模型会有突破，或者会有专门针对 CAD/EDA 场景微调的模型。到那时候再回来试，可能事半功倍。

关于系统架构：当前的 v3 Agent 架构是可复用的。Orchestrator + function calling + 原子工具这套模式不依赖具体的 EDA 场景。如果把工具集换一下，理论上可以做其他领域的 AI agent。

关于器件库：这块做得比较扎实，PDF 入库管线、器件数据模型、8 个设计知识字段，这些都是有价值的基础设施。如果有人想做电子元器件的结构化数据库，这块可以直接复用。


## AI 当前的能力局限性（个人观察）

通过这个项目我对 AI（特别是 LLM）的能力边界有了一些切身体会：

AI 擅长的事：
- 自然语言理解和意图提取（用户说什么器件什么参数，提取得很准）
- 结构化数据生成（给好 schema，AI 输出的 JSON 基本合规）
- 知识性问答（datasheet 里某个参数的含义、某种电路拓扑的工作原理）
- 代码审查和 bug 定位（给它看一段代码，找问题比人快）

AI 不擅长的事：
- 精确的数值计算（简单算术都会出错，复杂公式更不用说）
- 2D/3D 空间推理（坐标计算、碰撞检测、路径规划）
- 一致性维护（第一轮说 x=100，第二轮可能变成 x=110，它自己都不知道变了）
- 约束满足问题（同时满足"不重叠"、"不断开"、"美观"三个约束，它做不到）

这不是说 AI 没用。恰恰相反——我这个项目的 v3 架构就是把 AI 放在它擅长的位置（理解需求、做决策、选工具），把它不擅长的事交给本地代码。问题是渲染这块刚好踩中了 AI 最弱的领域。

另外一个感受是：prompt engineering 的天花板比想象中低。当任务本身超出模型的能力范围时，再怎么优化 prompt 都没用。prompt 能做的是把模型已有的能力更好地引导出来，但不能凭空创造能力。在 SVG 渲染这件事上，我花在 prompt 迭代上的时间远超花在写代码上的时间，但最终还是放弃了。

最后一点：AI coding agent（比如我用来开发这个项目的工具）效率确实很高，在理解需求、拆解任务、写样板代码方面帮了很大忙。但它也会"假装做完了"——改了个表面的东西就说"已修复"，实际上核心问题还在。所以在用 AI 做工程的时候，必须自己做验收，不能只看它说什么，要看它实际产出了什么。

## 关键收获

1. AI 做决策、代码做执行，这个分层原则是对的。但"决策"的边界要画清楚——空间布局虽然"看起来"是决策，实际上是计算。

2. 先想清楚验收标准再动手。我在 v1 和 v2 上浪费的时间，很大程度是因为没想清楚"什么算完成"就开始写代码。

3. task 拆分要到"可以验证"的粒度。"实现渲染"不是一个好 task，"Buck 电路的 Cin 和 Cout 不重叠且导线连到引脚端点"才是。

4. 不要怕推翻重来。v1 -> v2 -> v3 的重构看起来浪费了时间，但每次重做都因为更清楚问题在哪而做得更好。关键是每次推翻之前都要把"为什么当前方案不行"想明白，不要只是因为新方案看起来更酷。

5. 知道什么时候停。继续打磨 Buck 和 LDO 的渲染能让它更好看，但不改变"每种拓扑要手写布局"这个根本问题。识别出这种"天花板"并及时停手，比一直磨下去更重要。
